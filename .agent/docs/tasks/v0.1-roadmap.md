# v0.1 ロードマップ（概要）

## 目的
v0.1 の残課題を優先度順に整理し、短期リリース（v0.1.4 / v0.1.5）へ割り当てる。各リリースは小さな完了可能単位とし、テスト・仕様更新を同梱する。

## マイルストーン

- v0.1.4 (フォルダUI、レスポンシブ、i18n、R2画像アップロード)
  - 期間: 2〜3 週間
  - 主要項目:
    - フォルダ管理 UI の完成（作成/編集/階層表示） — API は実装済み
    - モバイルレスポンシブ対応（主要ページ）
    - i18n 初期セットアップ（日本語/英語）と文字列移行
    - R2 署名付きアップロード or Workers 経由アップロードの実装
    - a11y テスト + 必要な修正
    - `.agent/specs/` の更新（フォルダAPI、R2/アップロード仕様、i18n方針）

- v0.1.5 (統計ダッシュボード、デプロイ整備、パフォーマンス)
  - 期間: 2 週間
  - 主要項目:
    - 統計ダッシュボード（レシピ数・チェック履歴・タグ頻度）
    - バックエンドの集計API（Workersでの実装検討）
    - Pages + Workers のデプロイ手順とCI確認
    - パフォーマンス改善（Lighthouse 指標改善）
    - Chrome拡張準備（設計・manifest草案）

## 優先度と受け入れ基準

- 高優先度: フォルダUI、レスポンシブ、i18n、画像アップロード — 実用上必要な機能
- 中優先度: 統計ダッシュボード、デプロイ自動化、パフォーマンス改善
- 受け入れ基準:
  - 各機能にユニット/統合テストが追加され、CI を通ること
  - 変更が `.agent/specs/` に反映されていること（または spec-exception が記録されていること）
  - アクセシビリティの重大な違反が解消されていること

## 推奨作業フロー（各リリース）

1. トリアージ & 仕様確定（1-2 日） — QA に未決事項を登録し担当を決める
2. 実装（中規模タスクを 1 日〜3 日の小タスクに分割）
3. テスト追加（ユニット・統合・a11y）
4. `.agent/specs/` 更新と CHANGELOG 下書き
5. PR 作成 → レビュー → マージ → リリース
