## 【リリース】リリースワークフローの配置と CHANGELOG 更新要否
**質問**: `.github/workflows/release.yml` はどこに置くべきか。リリース PR に `CHANGELOG.md` の更新を必須化すべきか。

**回答**: リポジトリにワークフローを置き、`release/*` ブランチでテストして運用する。`main` に置いても問題ないが、`release/*` に限定しておくと実行が明確になる。リリース PR（`release/*`）では `CHANGELOG.md` の更新を必須とし、これを `Specs Check` ワークフローで検証する運用とします。

**決定日**: 2025-12-14

**関連**: commit 256d2ba (add release workflow)

## 【E2E】E2E スモークテストの実行ポリシー
**質問**: E2E スモークテストはすべての PR で回すべきか、それともリリースブランチ／定期実行のみにすべきか。

**回答**: 当面は CI コストを抑えるため、E2E スモークはリリースブランチまたはスケジュール実行で回す。CI リソースに余裕があれば、重要なフローに関して PR 実行を段階的に増やすことを検討します。

**決定日**: 2025-12-14

## 【a11y】カスタムコンポーネントへの ARIA 属性付与方針
**質問**: `UInput` や `UButton` のようなカスタム／サードパーティ要素に `aria-*` を付与する基準は？

**回答**: 可能な限り可視ラベルまたは `sr-only` ラベルを用いる。安易に `aria-*` をカスタム要素に付与するのは避け、その要素が正しいロールを公開している場合に限り ARIA を付与する。必要な場合はコンポーネント側の契約（role の公開など）を文書化する。

**決定日**: 2025-12-14

## 【仕様】同一 PR で仕様/ドキュメントが更新できない場合の運用
**質問**: コードや API 変更で同一 PR に仕様/ドキュメントを含められない場合、どうするか。

**回答**: PR に `spec-exception` ラベルを付け、`.agent/docs/qa/vX.Y-qa.md` に理由とフォローアップ日・担当者を記載する。`Specs Check` ワークフローはこのパターンを許容するが、QA エントリの作成を必須とする。

**決定日**: 2025-12-14
